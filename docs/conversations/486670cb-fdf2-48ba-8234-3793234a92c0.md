# Plan: Portal Steward Role + Portal Map + Documentation Constitution

## Context

The repo has two audiences (humans reading TechDocs, agents cold-starting from CLAUDE.md) and no owner for the space between them. The auditor diagnoses drift but cannot fix it. `docs/` mixes mirrors, narrative, and status with no labeling. Drift is already happening — `docs/agents/orchestrator.md` and `docs/architecture/NORTH_STAR.md` diverge from their sources.

This plan formalizes the missing owner: a portal-steward role that maintains the human-facing portal surface, keeps it aligned with machine truth, and never invents or changes source intent.

## Four-Layer Model

| Layer | What it is | Examples |
|-------|-----------|---------|
| 1. Source Truth | Architecture docs, agent roles, templates, sawmill artifacts, catalog | `architecture/NORTH_STAR.md`, `.claude/agents/builder.md`, `Templates/D1_CONSTITUTION.md` |
| 2. Machine Mirror | Exact or near-exact rendered copies in docs/ | `docs/architecture/NORTH_STAR.md`, `docs/agents/builder.md`, `docs/sawmill-templates/D1_CONSTITUTION.md` |
| 3. Human Narrative | Overview pages, summaries, "how this works" | `docs/index.md`, `docs/agent-onboarding.md`, `docs/dopejar-catalog.md` |
| 4. Audit / Status | Portal health, drift reports, operational state | `docs/PORTAL_STATUS.md`, `docs/status.md`, `sawmill/PORTAL_AUDIT_RESULTS.md` |

Rule: Narrative must never be the source of truth. It explains, summarizes, orients — it cannot silently diverge from the machine layer.

## Ownership Boundary

| Role | Writes to | Does not touch |
|------|-----------|---------------|
| portal-steward | `docs/*`, `mkdocs.yml`, `catalog-info.yaml`, `docs/PORTAL_MAP.yaml`, `docs/PORTAL_CONSTITUTION.md`, `docs/PORTAL_STATUS.md`, `sawmill/PORTAL_CHANGESET.md` | `architecture/*`, `.claude/agents/*` (source truth), `staging/*`, `.holdouts/*`, sawmill spec/build artifacts |

### **Claude** — 2026-03-07 16:31:56



This is very clarifying. The hook contract is actually **two valid approaches**, not one:

**Simple approach (what we already have):**
- `exit 2` + stderr message = block
- `exit 0` = allow

**Structured JSON approach:**
- `exit 0` + `{"hookSpecificOutput":{"permissionDecision":"deny",...}}` for PreToolUse
- `exit 0` + `{"decision":"block","reason":"..."}` for Stop

Both are correct. The guard and stop hook are using the simple `exit 2` approach, which **is documented and valid**. That's not a bug.

**The real bug is in the stop hook:** it doesn't check `stop_hook_active` from the input JSON. The Stop hook receives `{"stop_hook_active": false}` on first call, and `{"stop_hook_active": true}` on retry. If the hook blocks once and Claude tries to stop again, `stop_hook_active` will be `true` — and the hook **must** allow stopping to prevent an infinite loop. Our stop hook ignores this field entirely and uses a custom env var instead.

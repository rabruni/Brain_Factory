### **Claude** — 2026-03-17 21:03:11

Every mature build system hits this exact problem. Here's how they solved it:

**Make / CMake**
Variables defined once, referenced everywhere. `$(OUTDIR)/foo.o` — no rule ever hardcodes a path. Change `OUTDIR` once, everything follows. The build document describes what to build; the variable defines where.

**Helm / Kubernetes**
Charts never hardcode image tags or paths. `{{ .Values.image.tag }}` is a late-bound symbol. The chart is the spec. The values file is injected at deploy time. No generated artifact ever contains a literal path — only symbolic references that the runtime resolves.

**Terraform**
`aws_s3_bucket.my_bucket.arn` — you reference a resource by its symbolic name, never by its actual ARN. The ARN is resolved at `plan` time. The `.tf` file is the spec; Terraform is the resolver.

**12-Factor / Environment Variables**
Configuration — including paths — is never in source. Apps read `$DATA_DIR`. The runtime injects the value. The source is path-agnostic by design.

**The common pattern across all of them:**
- Authoring time: symbolic reference (what it IS)
- Runtime: resolution (where it LIVES)
- One resolver, one derivation path, no consumer ever independently derives the path

## What this means for Sawmill

The problem was **early binding**. The spec agent committed `staging/FMWK-900/` into D10 at Turn B time. That's authoring time. But the staging path is a runtime concern — only `run.sh` knows `$FMWK` authoritatively.

The fix is the same pattern every build system uses: D10 and BUILDER_HANDOFF are the spec (what to build). `{{STAGING_ROOT_PATH}}` in the Turn D prompt is the Helm values injection — late-bound, runtime-resolved, impossible to disagree with the verifier.

The documents become path-agnostic at authoring time. The runtime owns the path, always.

## Why Sawmills Were Built First

When settlers arrived on the frontier, the sawmill was one of the very first things they built — often before the church, the school, or the general store. The reason was simple: **you can't build anything else without lumber.** The sawmill was the prerequisite for everything. It turned raw forest into boards, beams, and planks. Without it, you had log cabins and dirt floors. With it, you had homes, barns, shops, furniture, fences — a town.

By 1840, there were nearly 32,000 sawmills in the United States. Where the settler went, the miller followed. The sawmill attracted carpenters, merchants, families. It was the seed of the community.

## How Safe Were They?

They were not safe at all. Amputations were common. There were no safety measures — no guards on the blades, no emergency stops, no training. The prevailing attitude was that each worker was responsible for their own safety. Management didn't see it as their job to make the environment safe. Injuries were considered the cost of doing business. The industry had one of the highest injury rates of the era, and the workers who ran the mills understood they were trading risk for the ability to build.

## The Sawmill — A Narrative

The sawmill exists to do one thing: turn a framework idea into tested, staged code. It takes a task in and produces a verified package out. Everything in between is the machine.

### How a piece of lumber enters the mill

It starts with a TASK.md file. This is the work order — it says which framework to build, what it owns, what it depends on, and what constraints apply. The orchestrator writes it based on the build plan and dependency graph. TASK.md is the log placed on the conveyor belt.

Before the log can move, the machine has to be set up. Three registries define how the sawmill operates: ROLE_REGISTRY.yaml says who does what work and which backend runs them. ARTIFACT_REGISTRY.yaml says what files get produced, where they go, and who owns them. PROMPT_REGISTRY.yaml says what each worker reads, what they're expected to produce, and how freshness is checked. These three registries are the gears, belts, and blade settings of the machine. run.sh reads them and configures itself.

### The five cuts

The log passes through five blades. Each one shapes it further.

**Cut 1 — Turn A (Spec Agent).** The spec agent picks up the work order and the authority documents — NORTH_STAR, BUILDER_SPEC, OPERATIONAL_SPEC, FWK-0, BUILD-PLAN. It reads the compressed templates for D1 through D6. Then it writes six documents: the constitution (D1), the specification (D2), the data model (D3), the contracts (D4), the research (D5), and the gap analysis (D6). The gate: D6 must have zero open items. If there are gaps, the log stops here.

**Cut 2 — Turn B (Plan Agent).** The same spec agent, fresh session, reads its own Turn A output — D1 through D6 — plus the planning templates. It produces four documents: the architecture plan (D7), the task breakdown (D8), the agent context handbook (D10), and the builder handoff (BUILDER_HANDOFF). These tell the builder exactly what to build, how to decompose the work, and what the testing strategy is. This is where the log gets measured and marked for cutting.

**Cut 3 — Turn C (Holdout Agent).** This runs in parallel with Turn B. A different agent — the holdout agent — reads ONLY D2 (specification) and D4 (contracts). Nothing else. No architecture, no plan, no data model. From just the spec and contracts, it writes acceptance test scenarios (D9) and puts them in `.holdouts/`, where the builder will never see them. This is the quality inspector who doesn't know how the lumber was cut — only what it's supposed to look like when it's done.

**Cut 4 — Turn D (Builder + Reviewer).** This is the main cut. It has three phases.

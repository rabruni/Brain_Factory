Skills from the `anthropics/skills` repo are installed by copying them into your Claude Code configuration directory. Here's how:

1. **Clone the repo:**
   ```bash
   git clone https://github.com/anthropics/skills.git
   ```

2. **Copy the skill(s) you want** into `~/.claude/skills/`:
   ```bash
   mkdir -p ~/.claude/skills
   cp -r skills/<skill-name> ~/.claude/skills/
   ```

   Each skill is a directory containing a `skill.md` file that defines triggers, instructions, and behavior.

3. **That's it** — Claude Code automatically picks up skills from `~/.claude/skills/` on next session.

You can also install them at the **project level** by placing them in `.claude/skills/` within a repo, which scopes them to that project only.

## What Maps to Sawmill

**Promptfoo** is the most directly useful. It lets you define test cases like "given this handoff, does the builder's output satisfy these constraints?" and run them programmatically. This maps to **Turn E (Evaluator)** — instead of a human-dispatched evaluator agent running holdout scenarios manually, Promptfoo could automate the evaluation loop. It also supports red-teaming, which maps to **Turn C (Holdout Agent)** — generating adversarial test cases that probe whether the builder's code handles edge cases the spec didn't anticipate.

**DeepEval's "faithfulness" and "hallucination" checks** map to a problem we've already hit: agents claiming things about documents they haven't read. The memory file records this as the most important operational lesson — "NEVER claim what a document says without quoting the exact line." DeepEval could mechanically verify that builder RESULTS.md claims are grounded in actual test output (faithfulness) rather than fabricated (hallucination).

**LangGraph's self-correction loops** map directly to **gap #4 (self-reflection)** from our analysis. Instead of just telling the builder "review your work with fresh eyes," you could wire a cyclic graph where the builder's output feeds back through a verification node before proceeding. This is what superpowers calls "self-reflection" but made mechanical.

**Claude PR Reviewer** could automate part of Turn E. Right now the evaluator is a fresh agent session — the PR reviewer action could run automatically on any PR from `staging/` branches.

## What Doesn't Map

**CrewAI / multi-agent orchestration** — sawmill already has this via `run.sh` and the turn structure. The isolation rules (builder never sees holdouts) are more rigorous than CrewAI's manager pattern. Adding CrewAI would add complexity without strengthening isolation.

**PyRIT / Garak** — these are for adversarial attacks on the LLM itself (jailbreaks, prompt injection). Sawmill's threat model is different: the agents aren't user-facing, they're constrained by the pipeline. The risk isn't prompt injection — it's drift. Garak won't catch an agent that slowly over-indexes on governance.

**MLflow** — designed for ML model evaluation at scale. Sawmill builds software, not ML models. The evaluation harness concept is useful but MLflow's specific tooling (metrics, experiment tracking) isn't a fit.

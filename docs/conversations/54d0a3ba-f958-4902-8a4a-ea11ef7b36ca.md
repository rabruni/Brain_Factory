## How Frameworks and Packages Work

**Framework Hierarchy** (Primitive #9, BUILDER_SPEC lines 193-226):

```
Framework (FMWK-NNN)
├── Spec Pack (SPEC-NNN)    ← architecture a builder reads
│   ├── Prompt Packs        ← governed LLM templates
│   ├── Module Packs        ← code
│   ├── Quality Measures    ← validation schemas
│   └── Protocol Definitions ← interface contracts
└── Files                   ← actual artifacts
```

**One framework = one package.** The package is the framework made installable.

**Package Lifecycle** (Primitive #8, BUILDER_SPEC lines 179-189):
1. **Author in staging** — builder works in `staging/FMWK-NNN/`. No governance. It's a workbench.
2. **Package in staging** — manifest.json with SHA256 hashes for every file, dependency declarations. Self-contained.
3. **Install through gates** — gate validation at each hierarchy layer. All pass → files copied to governed filesystem, ownership recorded, Ledger event appended. Any fail → entire package rejected. Atomic.
4. **Verify** — system-wide gate check. Inconsistent → rollback.

**Validation at every layer:** Framework declares valid governance? Spec pack references valid framework? Pack files match manifest? Hashes correct? File owned by exactly one pack? Logical hierarchy mirrors filesystem. Cryptographically verifiable chain.

**Frameworks are microservice-scoped:** single responsibility, explicit interfaces, own your data, declare dependencies, independently testable, contract-first.

## Agent Communication Model

**The filesystem is the message bus.** Agents never talk to each other directly. They communicate through files at known paths.

```
Orchestrator writes → sawmill/FMWK-NNN/TASK.md
Spec Agent reads TASK.md → writes D1-D6 → writes D7/D8/D10 + BUILDER_HANDOFF.md
Holdout Agent reads D2+D4 → writes .holdouts/FMWK-NNN/D9_HOLDOUT_SCENARIOS.md
Builder reads BUILDER_HANDOFF → writes staging/FMWK-NNN/** + RESULTS.md
Evaluator reads D9 + PR code → writes EVALUATION_REPORT.md + EVALUATION_ERRORS.md
```

Each agent cold-starts fresh — no memory of prior turns. It reads its role file, then reads the files the orchestrator tells it to read, in order. That's the entire context transfer.

## Isolation Walls

This is the governance teeth:

- **Builder NEVER sees** `.holdouts/` or D9 or evaluation reports
- **Evaluator NEVER sees** D1-D8, D10, handoff, RESULTS.md, builder reasoning

## How OthmanAdi/planning-with-files Enforces It

It uses **three hooks** — shell scripts that run automatically on tool events. The agent never opts in. The hooks fire whether the agent wants them to or not.

### Hook 1: PreToolUse — "Read the plan before every action"

```
matcher: "Write|Edit|Bash|Read|Glob|Grep"
command: cat task_plan.md | head -30
```

Every time the agent is about to use *any* tool, this fires. It dumps the first 30 lines of `task_plan.md` into the agent's context. The agent doesn't choose to read the plan — **the plan is injected into its attention window automatically**. This is the core trick. The agent can't drift because its goals are re-presented before every single action.

It never blocks (`{"decision": "allow"}` always). It's not a gate — it's a **forced context reload**.

### Hook 2: PostToolUse — "Remember to update the plan"

```
matcher: "Write|Edit"
command: echo '[planning-with-files] File updated. If this completes a phase, update task_plan.md status.'
```

After every file write, this injects a reminder into the agent's context. It's a nudge, not a block. But because it fires on every Write/Edit, the agent gets a persistent behavioral pressure to keep the plan file current.

### Hook 3: Stop — "You can't quit until the plan says you're done"

```bash
TOTAL=$(grep -c "### Phase" task_plan.md)
COMPLETE=$(grep -cF "**Status:** complete" task_plan.md)
if [ "$COMPLETE" -eq "$TOTAL" ]; then
    exit 0  # allow stop
else
    echo '{"followup_message": "Task incomplete (2/5 phases done). Read task_plan.md and continue."}'
fi
```

When the agent tries to stop, this counts phases vs. completed phases in the markdown file. If they don't match, it **auto-continues the agent** with a followup message. The agent literally cannot stop working until the plan file says all phases are complete. This is the hardest enforcement — it uses the `followup_message` JSON response to re-invoke the agent.

### What's NOT enforced

- The agent **can** write to any file — there's no tool restriction preventing it from editing code, specs, whatever. The hooks don't block writes.
- The agent **can** ignore the plan content injected by PreToolUse — it's context, not a constraint.
- There's no file-path allowlist. Nothing stops the agent from writing to files outside its lane.

### The enforcement model in one sentence

**Behavioral nudging via forced context injection, not hard access control.** The plan file is shoved into the agent's attention on every action, a reminder fires on every write, and the agent can't stop until the plan says it's done.
